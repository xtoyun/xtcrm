# Security Policy

## 支持版本

| 版本 | 支持状态 |
|------|----------|
| 0.13+ | ✅ 活跃支持 |
| < 0.13 | ❌ 不再维护 |

## 报告漏洞

**请勿通过公开 Issue 报告安全漏洞。**

如果你发现了安全漏洞，请通过以下方式私下报告：

- 邮箱：在 GitHub 个人主页（[xtoyun](https://github.com/xtoyun)）查找联系方式
- 微信：扫描 README 中的联系二维码，备注"XTCRM 安全报告"

### 报告应包含

1. 漏洞的详细描述
2. 复现步骤（尽可能具体）
3. 受影响版本
4. 可能的修复建议（如有）

### 处理流程

1. **确认收到**：24 小时内回复确认
2. **评估**：3 个工作日内完成评估，确定严重级别
3. **修复**：根据严重程度，尽快发布修复版本
4. **披露**：修复版本发布后公开披露（与报告者协商披露时间）
5. **致谢**：在发布说明中致谢报告者（除非报告者要求匿名）

### 严重级别

| 级别 | 示例 | 响应时效 |
|------|------|----------|
| **严重** | 未授权访问任意租户数据、SQL 注入、远程代码执行 | 48 小时内修复 |
| **高危** | 越权访问、身份伪造、敏感信息泄露 | 5 个工作日内修复 |
| **中危** | XSS、CSRF、不安全的默认配置 | 下个版本修复 |
| **低危** | 理论可攻击但利用条件苛刻 | 排入迭代计划 |

## 安全最佳实践

### 部署建议

1. **关闭调试模式**：生产环境 `.env` 中 `APP_DEBUG = false`
2. **限制 public/ 可写范围**：仅 `runtime/` 和 `uploads/` 需要写权限
3. **HTTPS 强制**：生产环境配置 SSL 证书
4. **Redis 密码**：生产环境必须设置 Redis 密码
5. **定期备份**：数据库和上传文件定期备份

### 开发建议

- 所有用户输入必须经过验证
- SQL 查询优先使用 Model（自带租户隔离 scope）
- API 权限通过 `checkAction()` 拦截
- 敏感操作写入 `crm_event_log` 审计日志
- 禁止在代码中硬编码密钥和密码
